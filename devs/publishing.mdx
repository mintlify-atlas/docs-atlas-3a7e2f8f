---
title: "Publishing your extension"
description: "Build, package, and distribute your CloudStream extension through a self-hosted repository that users can add in the app's settings."
---

CloudStream has no official plugin registry. Distribution is fully decentralized: you host a JSON manifest at any public URL, and users add that URL in the app. This page walks through the full workflow from building your `.cs3` file to keeping it up to date for your users.

## Step 1: Build the extension

<Steps>
  <Step title="Run the release build">
    From the root of your extension project, run:

    ```bash
    ./gradlew assembleRelease
    ```

    The output is an APK at `app/build/outputs/apk/release/`. Extension templates typically include a Gradle task that copies and renames this file to `<PluginName>.cs3`.
  </Step>

  <Step title="Sign the APK">
    For production releases, sign the APK with a keystore. Most extension templates configure a signing config in `build.gradle.kts`. For development and local testing, Android's debug signing is sufficient — you can sideload the file directly in **Settings → Extensions**.

    <Warning>
      Do not lose your keystore file. If you change or lose it, users will need to uninstall and reinstall your extension manually because Android's package verifier will reject updates signed with a different key.
    </Warning>
  </Step>

  <Step title="Verify the manifest">
    Every `.cs3` file contains a `manifest.json` in its assets, generated by the build plugin. It corresponds to the `BasePlugin.Manifest` class:

    ```json manifest.json (embedded in the .cs3)
    {
      "name": "MyPlugin",
      "pluginClassName": "com.example.MyPlugin",
      "version": 1,
      "requiresResources": false
    }
    ```

    | Field | Description |
    |---|---|
    | `name` | Display name of the plugin |
    | `pluginClassName` | Fully-qualified name of your `@CloudstreamPlugin` class |
    | `version` | Integer version; increment to trigger auto-updates |
    | `requiresResources` | Set to `true` if your plugin uses Android resources |

    Extension templates write this file automatically. If you are configuring Gradle manually, you need a task that writes `manifest.json` into `src/main/assets/` before the APK is assembled.
  </Step>
</Steps>

## Step 2: Create a repository manifest

A repository is a JSON file that CloudStream fetches to discover available extensions. It must be hosted at a publicly accessible URL.

### Repository structure

The top-level repository file corresponds to the `Repository` data class:

```json repo.json
{
  "name": "My Extension Repo",
  "description": "My personal CloudStream extensions",
  "manifestVersion": 1,
  "iconUrl": null,
  "pluginLists": [
    "https://raw.githubusercontent.com/username/repo/main/plugins.json"
  ]
}
```

| Field | Description |
|---|---|
| `name` | Repository display name shown in the app |
| `description` | Short description, shown in the repository browser |
| `manifestVersion` | Set to `1` |
| `iconUrl` | Optional URL to an icon image for the repository |
| `pluginLists` | Array of URLs, each pointing to a JSON file that lists individual plugins |

### Plugin list structure

Each URL in `pluginLists` points to a JSON array of plugin entries. Each entry corresponds to the `SitePlugin` data class:

```json plugins.json
[
  {
    "url": "https://raw.githubusercontent.com/username/repo/main/MyPlugin.cs3",
    "status": 1,
    "version": 1,
    "apiVersion": 1,
    "name": "My Plugin",
    "internalName": "MyPlugin",
    "authors": ["username"],
    "description": "Streams content from example.com",
    "repositoryUrl": null,
    "tvTypes": ["Movie", "TvSeries"],
    "language": "en",
    "iconUrl": null,
    "fileSize": null
  }
]
```

#### SitePlugin fields

| Field | Description |
|---|---|
| `url` | Direct download URL for the `.cs3` file |
| `status` | Availability status (see table below) |
| `version` | Integer version; any change triggers an auto-update for installed users |
| `apiVersion` | Set to `1` |
| `name` | Display name shown in the extensions browser |
| `internalName` | Stable internal identifier — **do not change this after publishing** |
| `authors` | List of author names |
| `description` | Short description shown in the extensions browser |
| `repositoryUrl` | Optional link to the extension's source repository |
| `tvTypes` | Content types this provider supports (e.g. `"Movie"`, `"TvSeries"`, `"Anime"`) |
| `language` | BCP 47 language tag for the provider's content (e.g. `"en"`, `"zh-TW"`) |
| `iconUrl` | Optional URL to an icon image for the extension |
| `fileSize` | File size in bytes; automatically generated by the Gradle build plugin |

#### Status codes

| Value | Meaning |
|---|---|
| `0` | Down — the provider is disabled and will not load |
| `1` | OK — normal operation |
| `2` | Slow — the provider works but may be unreliable |
| `3` | Beta only — visible only to users who have enabled beta extensions |

Use `status: 0` to remotely disable a broken provider without requiring users to uninstall it.

## Step 3: Host the repository

The simplest hosting option is a GitHub repository with the JSON files committed directly to the default branch.

<CardGroup cols={2}>
  <Card title="Hosting JSON on GitHub" icon="github">
    Commit `repo.json` and `plugins.json` to your repository. Use raw.githubusercontent.com URLs in your `pluginLists` array. GitHub serves raw files with no authentication required.
  </Card>
  <Card title="Hosting .cs3 files" icon="file">
    You can commit `.cs3` files directly to your repository (raw URLs work), or attach them as GitHub release assets and reference the asset download URL in your plugin list.
  </Card>
</CardGroup>

A minimal GitHub repository layout looks like this:

```
my-extensions/
├── repo.json          ← users add this URL to CloudStream
├── plugins.json       ← listed in repo.json pluginLists
└── MyPlugin.cs3       ← listed in plugins.json url field
```

### Adding a custom URL scheme

CloudStream also supports the `cloudstreamrepo://` scheme and `https://cs.repo/?` prefix as shorthand repository URLs. Both are resolved to a standard HTTPS URL before fetching. You can share either form with users.

## Step 4: Install for testing

Before sharing your repository URL publicly, test the full install flow:

1. Open CloudStream on your device.
2. Go to **Settings → Extensions → Add repository**.
3. Enter the URL to your `repo.json`.
4. Find your extension in the list and install it.

<Tip>
You can also sideload a `.cs3` file directly for faster iteration during development, without publishing it to a repository. Use **Settings → Extensions → Install from file** (or drag-and-drop in some versions).
</Tip>

## Step 5: Share your repository URL

Give users the direct URL to your `repo.json`. They add it in **Settings → Extensions → Add repository**. CloudStream fetches the manifest, resolves the plugin list URLs, and displays your extensions in the browser.

## Updating your extension

<Steps>
  <Step title="Increment the version">
    In your `plugins.json`, increment the `version` field for the updated plugin. CloudStream compares the installed version against the value in the plugin list and triggers an automatic update prompt when they differ.

    ```json plugins.json (updated)
    {
      "version": 2,
      ...
    }
    ```
  </Step>

  <Step title="Build and upload the new .cs3">
    Run `./gradlew assembleRelease` again and replace the `.cs3` file at the same URL. The `url` field in your plugin list must stay the same — it is what CloudStream downloads.
  </Step>

  <Step title="Commit and push">
    Push the updated `plugins.json` (and new `.cs3` if hosted in the repo). CloudStream fetches the plugin list fresh on each repository refresh, so users will see the update the next time they open the extensions screen.
  </Step>
</Steps>

<Warning>
Never change the `internalName` of a published plugin. CloudStream uses `internalName` to identify installed plugins and match them against repository entries. Changing it makes CloudStream treat the new entry as a different plugin entirely, leaving the old one orphaned on the user's device.
</Warning>

## Automating builds with GitHub Actions

You can automate the full build and release pipeline using GitHub Actions. A typical workflow:

1. On push to `main`, run `./gradlew assembleRelease`.
2. Rename the APK output to `<PluginName>.cs3`.
3. Update the `version` field in `plugins.json` (using `jq` or a custom script).
4. Commit the updated JSON and `.cs3` file back to the repository, or upload the `.cs3` as a release asset.

<Tip>
Search GitHub for CloudStream extension repositories to find example GitHub Actions workflows. Many maintainers of public extension collections have already solved the automation problem and share their workflow files openly.
</Tip>
